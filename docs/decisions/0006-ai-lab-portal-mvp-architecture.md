# 0006 AI Lab Portal MVP Architecture

Date: 2026-06-01

## Status

Accepted

## Context

`SPEC.md` defines a high-risk product: public publishing, admin authorization,
AI providers, crawlers, user-submitted URLs, raw external payload retention,
structured AI outputs, asynchronous jobs, and future provider integrations.

The repo currently contains Harness docs but no application implementation. The
first step is to convert the specification into living product contracts and
choose the MVP architecture before scaffolding app code.

## Decision

Accept the specification's MVP architecture direction:

- Next.js + TypeScript + Tailwind for public/admin frontend.
- FastAPI + Pydantic for backend API and structured validation.
- REST JSON APIs for MVP, with OpenAPI generated by FastAPI.
- PostgreSQL + pgvector as the primary data store and MVP embedding store.
- Redis + Celery for asynchronous crawl, AI, deduplication, and publishing jobs.
- S3-compatible object storage for large raw payloads, extracted content,
  images, and debug artifacts.
- Human approval required for public publishing in MVP.
- Start AI News with official/RSS/GitHub sources before X/Twitter.
- Treat crawled and submitted URLs as SSRF risk.
- Treat provider payloads and AI outputs as untrusted until schema-validated.

Authentication remains a follow-up decision before admin implementation. The
accepted candidates are Google Workspace OAuth or FastAPI-managed secure
HTTP-only sessions.

## Alternatives Considered

1. Single Next.js full-stack app.
   - Simpler deployment at first.
   - Rejected for MVP architecture because the product has heavy Python-native
     AI, crawler, worker, and data-processing workflows.
2. Add X/Twitter ingestion first.
   - Valuable intelligence source.
   - Rejected for early MVP because provider terms, rate limits, metrics,
     source quality, and cost risk are higher than official sources.
3. Separate vector database.
   - More specialized retrieval capability.
   - Rejected for MVP because PostgreSQL + pgvector is sufficient for initial
     semantic deduplication and avoids extra infrastructure.

## Consequences

Positive:

- The first build can focus on foundation and manual CMS before risky AI/social
  automation.
- Provider and AI boundaries remain testable and replaceable.
- The architecture itself supports the AI Lab showcase narrative.

Tradeoffs:

- Two application stacks require clear API contracts and local orchestration.
- Auth must be resolved before admin CRUD implementation.
- Celery/Redis/PostgreSQL make foundation heavier than a pure static site.

## Follow-Up

- Create MVP 0 foundation story packet.
- Record auth strategy decision before admin routes are implemented.
- Create source/provider contracts before Firecrawl, Apify, or X/Twitter work.
- Keep `SPEC.md` as input material; maintain living product truth in
  `docs/product/*`, story packets, decisions, and validation records.
