Architecture Comparison

The Celery path runs inline with task_always_eager=True and returns a JSON response. The Streaming path sends Server-Sent Events over HTTP. Each event is parsed client-side. Both paths use the same FastAPI router infrastructure and admin boundary authentication.

Latency Comparison by Stage

Bar chart comparing mean duration per stage. The streaming path includes HTTP SSE overhead, event serialization, and client-side parsing. The Celery path reflects inline FakeLLMService execution. Error bars show ±1 standard deviation across 5 runs.

Key Insight

The streaming path adds ~200ms of SSE infrastructure overhead regardless of stage complexity. This is a fixed cost — each SSE event requires HTTP framing, stream iteration, and JSON serialization.

The Celery path's overhead varies by stage (5–50ms) because the FakeLLMService processes different prompt types with different synthetic output sizes.

Time to First Byte (Streaming Only)

The TTFB is the time from POST to the first token SSE event. Since the fake stream uses await asyncio.sleep(0.001) before emitting the first token, this measures the combined HTTP + async scheduling overhead of the TestClient.

Interpretation & Next Steps

What This Means

  • The ~200ms overhead is the price of real-time — SSE gives you progressive token output at the cost of additional HTTP framing.
  • This overhead is dwarfed by actual LLM inference time (typically 5–30 seconds), making streaming's UX benefit effectively free in production.
  • The Celery path is leaner for fire-and-forget generation but offers no intermediate visibility.

Real-World Projection

  • With real LLM calls (5–15s): streaming overhead becomes 1–4% of total time.
  • With Runner.run_streamed() from Agents SDK: expect lower overhead since the SDK sends tokens directly without intermediate TestClient latency.
  • Recommendation: Run again with --real flag once an API key is available for production-accurate numbers.
